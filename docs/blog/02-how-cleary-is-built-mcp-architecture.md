# Part 2: How Cleary Is Actually Built (MCP, for Real)

*This is the technical half of the Cleary writeup. [Part 1](./01-the-problem-and-what-cleary-does.md)
covers what it does and why; this one covers how.*

The hackathon's required technology was MCP server integration, not "use an LLM," not
"build a Slack bot," specifically MCP. That constraint ended up being the best thing
that happened to this project's design, because it forced a real separation of concerns
instead of one big tangled Slack bot file.

## Three pieces, three jobs

Cleary is three genuinely independent pieces talking over standard protocols:

```
Slack side panel  (HOST)
        |  Bolt Events / Web API
        v
agent/  (CLIENT)  -- Gemini 2.5 Flash reasons here
        |  MCP over stdio
        v
cleary-mcp/  (SERVER)  -- four pure-function accessibility tools
```

**Slack is the host.** It renders the chat UI, the streaming task-card steps, and the
Block Kit buttons. It has zero opinions about accessibility checking.

**The Bolt agent is the client.** This is a Python process using
[Bolt for Python](https://slack.dev/bolt-python/), connected to Slack over Socket Mode
(an outbound WebSocket, no public URL needed for local dev). Its entire job is running
the "agent loop": take the user's message, ask an LLM what to do, notice when the LLM
wants a tool, go get that tool's result, hand it back, repeat.

**`cleary-mcp` is the server**, and it is boring on purpose. It's a
[`FastMCP`](https://github.com/modelcontextprotocol/python-sdk) process exposing four
plain Python functions as tools. It takes text or Slack Block Kit data in, and returns
plain JSON-serializable dicts out. It doesn't import anything Slack-specific. You could
swap the Slack host for a Discord bot or a CLI tool tomorrow, and this file would not
need to change by a single line.

## The tool-calling loop, concretely

Here's roughly what happens on every user turn, from `mcp_client.py`:

```python
async with stdio_client(_SERVER_PARAMS) as (read, write):
    async with ClientSession(read, write) as session:
        await session.initialize()
        gemini_tools = await _mcp_tools_to_gemini_tools(session)

        for _ in range(MAX_TOOL_ROUNDS):
            response = client.models.generate_content(
                model=MODEL,
                contents=contents,
                config=types.GenerateContentConfig(
                    system_instruction=SYSTEM_PROMPT,
                    tools=gemini_tools,
                ),
            )
            # ...inspect response for function_call parts, execute them via
            # session.call_tool(name, args), feed results back, loop again
```

The client starts the MCP server as a subprocess, over stdio (standard input/output,
the simplest possible transport, just two processes talking through pipes). It calls
`list_tools()` to discover what's available, converts each tool's JSON Schema into the
shape Gemini's function-calling API expects, and hands all of that to the model
alongside the user's message.

Gemini decides, on its own, whether it needs a tool. If it does, the client executes it
against the live subprocess, wraps the result as a "function response," and sends the
whole conversation back to Gemini for another round. This repeats until Gemini stops
asking for tools and just answers.

None of this loop logic is accessibility-specific. It's generic MCP client machinery.
The interesting part, from an "agent for good" perspective, lives one layer up: in how
the *results* of that loop get shown to a human.

## Making the invisible visible: task cards and impact lines

Every tool call streams a visible step into the Slack UI, "Calling readability_score...",
then "Called readability_score" with its actual output, using Slack's
`TaskUpdateChunk` streaming primitive. If a tool call fails, that step shows an
`error` state instead of silently swallowing the problem.

One deliberate choice: the "impact line" you see (like *"reading grade 45.6, aim for
grade 8 or below"*) is **not generated by the LLM**. It's computed directly from a
tool's structured JSON output, in a small pure function called `_impact_line()`. That
guarantees the number shown is always accurate and always present, instead of depending
on the model remembering to mention it in its own words. The model handles the
explanation and the rewrite; the numbers are handled deterministically.

## Human-in-the-loop, enforced structurally

The system prompt tells Gemini: when you propose an actual rewrite, wrap only the
rewritten text between `<<<REWRITE>>>` and `<<<END REWRITE>>>` markers. The client
regex-extracts that span, strips the markers from what's displayed (so nobody sees
literal marker tokens), and separately hands the raw rewrite text up to the message
handler.

From there, `message.py` attaches a Block Kit `ActionsBlock` with two buttons, **Copy
rewrite** and **Post rewrite**, each carrying the rewrite text as their button value. The
action handlers behind those buttons are the *only* code paths in the entire app that
can post a rewrite anywhere, and they only run when a Slack user physically clicks a
button. The LLM never gets a code path that lets it call `chat.postMessage` with a
rewrite directly. That's not a prompt instruction that could be ignored; it's a
structural guarantee.

## The one big pivot: swapping the LLM mid-build

Cleary's design doc originally specified Claude as the reasoning model. Partway through,
we hit an Anthropic billing limit on a fresh account and didn't want to pay for a
hackathon project, so we switched to Gemini 2.5 Flash on Vertex AI, authenticated via
`gcloud` credentials instead of an API key.

What's notable is *how little that changed*. The MCP server, the four tools, the
task-card UI, the rewrite-button logic, none of it moved. The actual rewrite was almost
entirely in one function: converting MCP's JSON-Schema tool definitions into whatever
shape the new model's function-calling API expects. That's kind of the point of MCP:
the protocol between client and server is the stable interface. The LLM sitting on the
"client" side making decisions is swappable underneath it.
